Micro-benchmarking code always looks simple: a few variables, a small for loop and two std::chrono calls. I think this simplicity is an illusion and that, most of the time, micro-benchmarking code is either complicated or inaccurate.

You have to be careful that the compiler does not optimize out the code you are timing, all the caches are usually hot (as you loop over the same operation) and you will get a totally different cache hit rate than in production, a “production-like” set of inputs has to be generated — without affecting the resuts of the benchmark of course! —, on each iteration of the benchmark the state of your object is changing but should still allow the benchmark to continue, etc. It is really hard.

And this is why I usually don’t micro-benchmark: it is too hard for what it is ; you will get numbers and finally be disappointed or puzzled because these numbers in production will be different.

I prefer measuring the impact of my code changes directly in production — or in a production-like environment. The initial cost of setting up such timing system might be higher, but on the long term you get a lot of benefits from it. Lots of code changes can simply not be micro-benchmarked at all. In his talk Optimization Tips - Mo’ Hustle Mo’ Problems, Andrei Alexandrescu is presenting the speed gain of not inlining the destructor of an object — by simply defining an empty destructor in the source file instead of the default one. Good luck to micro-benchmark that, and if you actually do it somehow, you will get the opposite result, as having the call inlined will be of course faster in a benchmark but in the context of your application, slower — mainly due to the instruction cache.

This post is about my last adventure with a micro-benchmark.

The tragedy

At some point, you decide to re-order the function calls. You don’t know why — just for convenience, because it is nicer like that. And boom. You didn’t expect anything to change because you were staring these numbers for hours… but you now get:

100000 iterations of umap<int, int>::emplace() took 1143ms

How come? You run again and get the same results. Nothing changed. You checkout the previous commit and test again: it is fixed. How can a simple re-ordering of function calls change by more than a factor two the time of this for statement?

Hardware counters are your friends

In this kind of scenario, you have a quite limited set of options.

First of all, you might think about looking at the disassembly. I know that we all love the GCC explorer, but when you are benchmarking containers, the assembly code will be very long. Also, reodering function calls is not a small change, so the entire assembly code will have changed and it will hard to compare that by hand.

The most powerful tool you could use in this case if your intuition does not guide you towards the light is a feature offered by your CPU, called hardware counters. perf is the simplest way to use them: 

$ perf stat -e cycles,instructions,branches,branch-misses,cache-misses ls

    Performance counter stats for 'ls':

    1 783 212 cycles                   
    1 246 362 instructions       #    0,70  insns per cycle        
      253 533 branches                                                    
       12 550 branch-misses      #    4,95% of all branches        
        6 770 cache-misses                                                

     0,001311920 seconds time elapsed

As the name indicates, hardware counters are CPU specific. The most classic ones — about instructions, cycles, branches and instruction/data caches — are available on most of the Intel chips though.

In order to use them, the reference is the libpapi. As it is a C library, I wrote a while ago a small C++ header file libpapipp to wrap it.

Heap fragmentation

The cause for this difference of time is simply the heap. I maintain a simple C++ wrapper around the malloc hooks that gives to us the immediat answer here:

benchmark_unordered_map_emplace() took 565ms
time spent in malloc(): 231ms
time spent in free(): 1ms
time spent in realloc(): 0ms
benchmark_unordered_map_emplace() took 1097ms
time spent in malloc(): 916ms
time spent in free(): 94ms
time spent in realloc(): 0ms
